**Read 01-01-SUMMARY.md first** to determine which Sonos library won the spike. All code below must use the winning library. The patterns below show both options — implement ONLY the one that passed the spike.

**1. Create `src/services/discovery.ts`:**

Define and export a `discoverSpeakers` function that:
- **Strategy: SSDP first, manual IP fallback.**
- Attempts SSDP discovery with a 5-second timeout (see research: keep timeout low for startup, pitfall 5)
- If SSDP finds speakers, return them
- If SSDP finds zero speakers OR throws an error, log a warning and attempt manual IP fallback
- Manual IP fallback: parse `SONOS_SPEAKER_IPS` env var (comma-separated), initialize from each IP
- If both fail and no speakers found, log an error clearly: "No Sonos speakers found. Check network or set SONOS_SPEAKER_IPS." — do NOT throw, return empty array (the app should start in degraded mode, not crash)
- Log each discovered speaker: name, IP, UUID (informative logging per user decision)

For `@svrooij/sonos`:
- Use `SonosManager.InitializeWithDiscovery(5)` for SSDP
- Use `SonosManager.InitializeFromDevice(firstIp)` for manual fallback (this discovers all devices from one seed IP)

For `sonos` (bencevans):
- Use `new AsyncDeviceDiscovery()` with `discoverMultiple({ timeout: 5000 })`
- For manual fallback, create `new Sonos(ip)` for each IP

For direct SOAP:
- Use raw dgram M-SEARCH for SSDP (see research pattern 4)
- For manual fallback, query `GetZoneGroupState` via fetch to `http://{ip}:1400/ZoneGroupTopology/Control`

**2. Create `src/services/registry.ts`:**

Define and export a `SpeakerRegistry` class:
- Internal storage: `Map<string, SpeakerInfo>` where key is speaker UUID
- `SpeakerInfo` type: `{ uuid: string, name: string, ip: string, groupName: string, isCoordinator: boolean, coordinatorUuid: string }`
- Methods:
  - `populate(devices)`: Takes array of discovered devices, extracts info, stores in map. For each device, determine if it is the coordinator of its zone group (see research pattern 3 — MUST identify coordinators correctly)
  - `getAll()`: Returns array of all SpeakerInfo
  - `getById(uuid)`: Returns SpeakerInfo or undefined
  - `getCoordinator(uuid)`: Given any speaker UUID, returns the SpeakerInfo of its zone group coordinator. This is CRITICAL — commands must go to the coordinator (research anti-pattern: sending to wrong speaker)
  - `clear()`: Empties the registry (for re-discovery)
  - `count`: Getter returning speaker count

**3. Create `src/plugins/sonos.ts`:**

Fastify plugin (using `fastify-plugin` to break encapsulation) that:
- Imports `discoverSpeakers` from services/discovery
- Imports `SpeakerRegistry` from services/registry
- On registration:
  1. Calls `discoverSpeakers()` passing config (manual IPs from `fastify.config.SONOS_SPEAKER_IPS`)
  2. Creates a `SpeakerRegistry` and calls `populate()` with discovered devices
  3. Logs: `"Speaker registry: {N} speakers in {M} zone groups"`
  4. Decorates Fastify with `fastify.speakers` (the registry) and `fastify.sonosDevices` (the raw device objects/manager for sending commands)
- Add TypeScript type augmentation so `fastify.speakers` and `fastify.sonosDevices` are typed on the Fastify instance
- Keep the raw Sonos library device references accessible (needed by the command endpoint to call play/pause/volume on actual devices)

**4. Update `src/app.ts`:**
- Import and register `sonosPlugin` AFTER `envPlugin` but BEFORE routes
- Import and register `speakerRoutes` (from next task's file — create the import but the file will be created in Task 2)
- Registration order: envPlugin → sonosPlugin → healthRoutes → speakerRoutes

Create `src/routes/speakers.ts` as a Fastify route plugin:

**GET /speakers**
- Returns JSON array of all speakers from the registry (`fastify.speakers.getAll()`)
- Response shape per speaker:
  ```json
  {
    "uuid": "RINCON_xxx",
    "name": "Living Room",
    "ip": "192.168.1.10",
    "groupName": "Living Room",
    "isCoordinator": true,
    "coordinatorUuid": "RINCON_xxx"
  }
  ```
- Add Fastify JSON schema for the response (for documentation and validation)
- If registry is empty (no speakers discovered), return `200` with empty array `[]` — not a 404 or 500

**POST /speakers/:id/command**
- `:id` is the speaker UUID
- Request body JSON schema: `{ command: string, value?: number }`
- Supported commands: `play`, `pause`, `stop`, `next`, `previous`, `volume` (value: 0-100), `mute`, `unmute`
- **CRITICAL: Command routing to coordinator.**
  - For transport commands (play, pause, stop, next, previous): resolve the coordinator using `fastify.speakers.getCoordinator(id)` and send the command to the coordinator device
  - For rendering commands (volume, mute, unmute): send directly to the target speaker (volume is per-speaker, not per-group)
  - Log: `"Command {command} sent to {speakerName} (coordinator: {coordinatorName})"`
- Error handling:
  - Speaker not found: 404 `{ error: "Speaker not found", uuid: id }`
  - Unknown command: 400 `{ error: "Unknown command", command: body.command }`
  - Command failure (Sonos error): 502 `{ error: "Command failed", detail: error.message }`
- Success response: 200 `{ ok: true, command: body.command, speaker: speakerName }`

**Implementation notes by library:**

For `@svrooij/sonos`:
- Access raw SonosDevice from `fastify.sonosDevices` (the SonosManager)
- Transport: `device.AVTransportService.Pause()`, `.Play()`, `.Next()`, `.Previous()`, `.Stop()`
- Volume: `device.RenderingControlService.SetVolume({ InstanceID: 0, Channel: 'Master', DesiredVolume: value })`
- Mute: `device.RenderingControlService.SetMute({ InstanceID: 0, Channel: 'Master', DesiredMute: true/false })`

For `sonos` (bencevans):
- `device.pause()`, `device.play()`, `device.next()`, `device.previous()`, `device.stop()`
- `device.setVolume(value)`, `device.setMuted(true/false)`

For direct SOAP:
- Send SOAP envelopes to `http://{ip}:1400/MediaRenderer/AVTransport/Control` for transport
- Send to `http://{ip}:1400/MediaRenderer/RenderingControl/Control` for volume/mute
- Include correct SOAPAction headers
