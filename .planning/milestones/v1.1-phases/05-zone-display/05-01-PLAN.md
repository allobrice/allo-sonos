# Plan 05-01: Zone Store (Pinia) + WebSocket Integration

## Goal

Create a `useZonesStore` Pinia store that holds the full list of `ZoneState` objects, populated via WebSocket snapshot on connect and kept live via `state_changed` events. The WebSocket composable is extended to support message dispatch. No UI is built in this plan.

## must_haves

- `useZonesStore` exposes a `zones` array (reactive, ordered by zone name) populated on WebSocket `snapshot` event — RT-01
- `state_changed` events are applied to the correct zone by UUID — RT-02
- WebSocket reconnects automatically with exponential backoff (1s → 2s → 4s → 8s cap) — RT-03
- Store is usable independently by ZoneCard components in plan 05-02

## Context

**Backend WebSocket protocol (from `/backend/src/routes/ws.ts` and `state-cache.ts`):**

On connect, the server immediately sends:
```json
{ "event": "snapshot", "data": [ZoneState, ZoneState, ...] }
```

On any zone state change:
```json
{ "event": "state_changed", "data": ZoneState }
```

**ZoneState shape (from `state-cache.ts`):**
```ts
interface ZoneState {
  uuid: string
  name: string
  playState: string        // 'PLAYING' | 'PAUSED_PLAYBACK' | 'STOPPED' | 'TRANSITIONING' | 'UNKNOWN'
  volume: number           // 0-100
  muted: boolean
  title: string | null
  artist: string | null
  album: string | null
  source: string | null    // 'spotify' | 'deezer' | 'tunein' | 'library' | null
  reachable: boolean
  lastSeen: number
}
```

**Existing WebSocket composable:** `frontend/src/composables/useWebSocket.ts`
- Currently manages connection, reconnect, and exposes `connected` ref
- Used by `AppHeader.vue` for the live indicator
- Does NOT yet dispatch parsed messages

**Design tokens available:** `var(--color-*)`, `var(--space-*)`, `var(--radius-*)`, `var(--shadow-card)`

### Extend useWebSocket composable to support message handlers

Modify `frontend/src/composables/useWebSocket.ts` to:

1. Accept an optional `onMessage` callback parameter: `onMessage?: (event: string, data: unknown) => void`
2. In `ws.onmessage`, parse the JSON payload and call `onMessage(parsed.event, parsed.data)` if provided
3. Preserve all existing behavior — `connected` ref, reconnect logic, `onMounted`/`onUnmounted` lifecycle — unchanged

**Signature change:**
```ts
export function useWebSocket(onMessage?: (event: string, data: unknown) => void) {
```

**In `ws.onmessage`:**
```ts
ws.onmessage = (msg) => {
  try {
    const { event, data } = JSON.parse(msg.data)
    onMessage?.(event, data)
  } catch {
    // ignore malformed messages
  }
}
```

`AppHeader.vue` calls `useWebSocket()` with no arguments — must remain compatible (no breaking change).
</task>

<task id="3" wave="1">
### Implement useZonesStore Pinia store

Add the store implementation to `frontend/src/stores/zones.ts` (same file as the type):

```ts
import { ref, computed } from 'vue'
import { defineStore } from 'pinia'
import type { ZoneState } from './zones'  // self-referential — or just define inline

export const useZonesStore = defineStore('zones', () => {
  const _zones = ref<Map<string, ZoneState>>(new Map())

  // Sorted by zone name ascending for stable, predictable grid order
  const zones = computed<ZoneState[]>(() =>
    Array.from(_zones.value.values()).sort((a, b) => a.name.localeCompare(b.name))
  )

  function applySnapshot(data: ZoneState[]) {
    _zones.value = new Map(data.map(z => [z.uuid, z]))
  }

  function applyStateChanged(data: ZoneState) {
    _zones.value.set(data.uuid, data)
    // Trigger reactivity — replace the Map reference
    _zones.value = new Map(_zones.value)
  }

  return { zones, applySnapshot, applyStateChanged }
})
```

**Reactivity note:** Pinia ref holding a Map requires explicit Map reference replacement (`_zones.value = new Map(_zones.value)`) to trigger computed recalculation. Do not mutate in place.
</task>

<task id="4" wave="1">
### Wire store into ZonesView with WebSocket message dispatch

Modify `frontend/src/views/ZonesView.vue` to:

1. Import and call `useZonesStore()`
2. Import `useWebSocket` with an `onMessage` handler that dispatches `snapshot` and `state_changed` events to the store
3. Show a loading skeleton while `zones.length === 0` and the WebSocket is not yet connected (use `connected` from `useWebSocket`)

```vue
<script setup lang="ts">
import { useZonesStore } from '@/stores/zones'
import { useWebSocket } from '@/composables/useWebSocket'

const store = useZonesStore()

const { connected } = useWebSocket((event, data) => {
  if (event === 'snapshot') store.applySnapshot(data as any)
  else if (event === 'state_changed') store.applyStateChanged(data as any)
})
</script>

<template>
  <div class="zones-view">
    <!-- Loading skeleton: shown while no zones loaded yet -->
    <div v-if="store.zones.length === 0" class="zones-skeleton">

    <!-- Zone grid: rendered once zones are available -->
    <!-- ZoneCard components added in plan 05-02 -->
    <div v-else class="zones-grid">
      <div v-for="zone in store.zones" :key="zone.uuid" class="zone-placeholder">
        {{ zone.name }}
      </div>
    </div>
  </div>
</template>
```

**Note on AppHeader:** `AppHeader.vue` currently calls `useWebSocket()` with no arguments to get `connected`. After the composable change in Task 2, this continues to work unchanged (no `onMessage` provided = no dispatch). However, this means there are now two separate WebSocket connections (one in AppHeader, one in ZonesView). This is correct for the current architecture — the WS indicator in the header manages its own connection lifetime.

**Alternative approach if two connections are undesirable:** Hoist the `useWebSocket` call to `App.vue` and pass `connected` down as a prop to `AppHeader`. This is left to the executor's judgment — either approach is acceptable.
