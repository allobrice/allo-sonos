  <action>
Create the `backend/` directory as the project root for the Node.js backend.

**1. Initialize the project:**
```bash
cd backend && npm init -y
```

**2. Install dependencies:**
```bash
npm install fastify @fastify/env fastify-plugin
npm install -D typescript @types/node tsx
```

**3. Create `tsconfig.json`:**
- `target`: `ES2022` (avoids Fastify deprecation warnings — see research pitfall 4)
- `module`: `NodeNext`, `moduleResolution`: `NodeNext`
- `outDir`: `dist`, `rootDir`: `src`
- `strict`: true, `esModuleInterop`: true, `skipLibCheck`: true

**4. Create `src/plugins/env.ts`:**
- Use `@fastify/env` with `fastify-plugin` to break encapsulation
- Schema validates:
  - `SONOS_PIN`: string, required, minLength 4, maxLength 4
  - `SONOS_SPEAKER_IPS`: string, default `''` (comma-separated IPs)
  - `PORT`: integer, default 3000
- Export as default using `fp()` wrapper
- Do NOT use `dotenv: true` — config comes from docker-compose environment only (per user decision)

**5. Create `src/routes/health.ts`:**
- `GET /health` returns `{ status: 'ok' }`
- Export as a standard Fastify route plugin

**6. Create `src/app.ts`:**
- Import and create Fastify instance with logger configured:
  - level: `'info'`
  - In dev (NODE_ENV !== 'production'): use `pino-pretty` transport (install as dev dep: `npm install -D pino-pretty`)
  - In production: default JSON output
- Register plugins in order: envPlugin first, then healthRoutes
- Export `buildApp()` async function that returns the configured instance

**7. Create `src/server.ts`:**
- Import `buildApp` from `./app`
- Call `buildApp()`, then `fastify.listen({ port: fastify.config.PORT, host: '0.0.0.0' })`
- Add graceful shutdown: listen for SIGTERM and SIGINT, call `fastify.close()`
- On error, log and `process.exit(1)`

**8. Add npm scripts to `package.json`:**
- `"dev": "tsx watch src/server.ts"`
- `"build": "tsc"`
- `"start": "node dist/server.js"`

**9. Create `Dockerfile` (multi-stage):**
- Builder stage: `node:22-alpine`, copy package*.json, `npm ci`, copy tsconfig.json + src, `npm run build`
- Production stage: `node:22-alpine`, copy package*.json, `npm ci --omit=dev`, copy dist from builder, `EXPOSE 3000`, `CMD ["node", "dist/server.js"]`

**10. Create `docker-compose.yml`:**
- Service `backend`: build `.`, `network_mode: host`, `restart: always`
- Environment: `NODE_ENV=production`, `PORT=3000`, `SONOS_PIN=1234`, `SONOS_SPEAKER_IPS=`
- No `ports:` mapping (incompatible with host network mode — see research anti-patterns)

**11. Create `.dockerignore`:**
- `node_modules`, `dist`, `.git`, `*.md`, `spike-*`

**TypeScript type augmentation:** Add Fastify config type declaration so `fastify.config.PORT`, `fastify.config.SONOS_PIN`, etc. are typed. This can be in `env.ts` or a separate `types.d.ts` — use your judgment.

  <action>
Create `backend/spike-sonos.ts` — a standalone script that tests Sonos library compatibility on real hardware. This is the critical risk gate for the entire phase.

**Install both spike candidates:**
```bash
cd backend && npm install @svrooij/sonos sonos
```

**Spike script structure:**

The script accepts a CLI argument to select the library: `tsx spike-sonos.ts svrooij` or `tsx spike-sonos.ts bencevans`.

**Test 1: SSDP Discovery**
- `@svrooij/sonos`: Call `new SonosManager()` then `manager.InitializeWithDiscovery(5)` (5s timeout). Log device count and each device's Name, GroupName, uuid, host/IP.
- `sonos` (bencevans): Use `AsyncDeviceDiscovery` to discover devices with 5s timeout. Log each device's host, currentTrack if available.

**Test 2: Manual IP Initialization**
- Accept an optional `--ip=192.168.x.x` flag
- `@svrooij/sonos`: Call `manager.InitializeFromDevice(ip)`. Log all devices found.
- `sonos` (bencevans): Create `new Sonos.Sonos(ip)`. Call `device.getAllGroups()` and log groups.

**Test 3: Basic Command**
- On the first discovered device (or the coordinator of the first group), send a `pause()` command.
- Log success or failure with the error message.
- Then send a `play()` command to restore state.

**Test 4: Zone Group Coordinator Identification**
- `@svrooij/sonos`: Log each device's coordinator status (GroupName matching, coordinator property).
- `sonos` (bencevans): Call `getAllGroups()`, log each group's coordinator and members.
- Direct SOAP fallback: If both libraries fail tests 1-3, include a raw SOAP section that sends `GetZoneGroupState` to port 1400 (using Node.js native `fetch`) and parses the XML response. Use `fast-xml-parser` if needed (`npm install fast-xml-parser`).

**Output format:** The script should print a clear summary:
```
=== SPIKE RESULTS ===
Library: @svrooij/sonos
SSDP Discovery: PASS/FAIL (N devices)
Manual IP Init: PASS/FAIL (N devices)
Basic Command:  PASS/FAIL (error message if fail)
Coordinator ID: PASS/FAIL (coordinator name/uuid)
=== RECOMMENDATION ===
Use: @svrooij/sonos | sonos (bencevans) | direct SOAP
```

**Important:** This script runs on the developer's machine (not in Docker) against real Sonos speakers on the local network. It must be runnable with `npx tsx spike-sonos.ts svrooij` or `npx tsx spike-sonos.ts bencevans`.

If BOTH libraries fail all tests, the script should attempt the direct SOAP fallback (GetZoneGroupState via fetch to port 1400) and report whether raw HTTP works. This determines whether to proceed with a library or implement thin SOAP wrappers.
