Create `backend/src/services/sonos-state.ts` — a new module that reads the current state of a Sonos speaker via three parallel SOAP GET calls. This is the genuinely new code in Phase 2 (Phase 1 only wrote commands, never read state back).

**Export the following:**

1. `SpeakerState` interface:
```typescript
export interface SpeakerState {
  playState: string   // Raw Sonos value: "PLAYING", "PAUSED_PLAYBACK", "STOPPED", "TRANSITIONING", "NO_MEDIA_PRESENT"
  volume: number      // 0-100
  muted: boolean      // true/false (coerced from Sonos 0/1)
}
```

2. `readSpeakerState(targetIp: string, coordinatorIp: string): Promise<SpeakerState | null>` — main export used by route handlers. Runs three SOAP reads in parallel via `Promise.all`. Returns `null` on ANY error (user decision: state reading is best-effort).

**Internal (non-exported) SOAP read functions:**

Create a **single shared** `XMLParser` instance at module level with `{ removeNSPrefix: true }` (CRITICAL — see Pitfall 1 in research). This strips namespace prefixes so response keys are `Envelope > Body > GetTransportInfoResponse > CurrentTransportState` (not `s:Envelope > s:Body > u:GetTransportInfoResponse`).

```typescript
import { XMLParser } from 'fast-xml-parser'
const parser = new XMLParser({ removeNSPrefix: true })
```

3. `soapGetTransportInfo(coordinatorIp: string): Promise<string>` — internal function.
   - SOAP URL: `http://${coordinatorIp}:1400/MediaRenderer/AVTransport/Control`
   - SOAPAction: `"urn:schemas-upnp-org:service:AVTransport:1#GetTransportInfo"`
   - SOAP body: `<u:GetTransportInfo xmlns:u="urn:schemas-upnp-org:service:AVTransport:1"><InstanceID>0</InstanceID></u:GetTransportInfo>`
   - Parse response: `parsed?.Envelope?.Body?.GetTransportInfoResponse?.CurrentTransportState ?? 'UNKNOWN'`
   - NOTE: Read transport state from COORDINATOR IP (same routing as transport write commands)

4. `soapGetVolume(targetIp: string): Promise<number>` — internal function.
   - SOAP URL: `http://${targetIp}:1400/MediaRenderer/RenderingControl/Control`
   - SOAPAction: `"urn:schemas-upnp-org:service:RenderingControl:1#GetVolume"`
   - SOAP body: `<u:GetVolume xmlns:u="urn:schemas-upnp-org:service:RenderingControl:1"><InstanceID>0</InstanceID><Channel>Master</Channel></u:GetVolume>`
   - Parse response: `Number(parsed?.Envelope?.Body?.GetVolumeResponse?.CurrentVolume ?? 0)`
   - NOTE: Read volume from TARGET IP (per-speaker rendering)

5. `soapGetMute(targetIp: string): Promise<boolean>` — internal function.
   - SOAP URL: `http://${targetIp}:1400/MediaRenderer/RenderingControl/Control`
   - SOAPAction: `"urn:schemas-upnp-org:service:RenderingControl:1#GetMute"`
   - SOAP body: `<u:GetMute xmlns:u="urn:schemas-upnp-org:service:RenderingControl:1"><InstanceID>0</InstanceID><Channel>Master</Channel></u:GetMute>`
   - Parse response: `raw === 1 || raw === '1'` (Pitfall 3 — CurrentMute returns integer 0/1, not boolean)
   - NOTE: Read mute from TARGET IP (per-speaker rendering)

**readSpeakerState implementation:**
```typescript
export async function readSpeakerState(
  targetIp: string,
  coordinatorIp: string,
): Promise<SpeakerState | null> {
  try {
    const [playState, volume, muted] = await Promise.all([
      soapGetTransportInfo(coordinatorIp),
      soapGetVolume(targetIp),
      soapGetMute(targetIp),
    ])
    return { playState, volume, muted }
  } catch {
    return null  // Best-effort: user decision says state failure != command failure
  }
}
```

**SOAP envelope pattern for reads** — same structure as write envelopes but with GET-style actions:
```xml
<?xml version="1.0"?>
<s:Envelope xmlns:s="http://schemas.xmlsoap.org/soap/envelope/"
            s:encodingStyle="http://schemas.xmlsoap.org/soap/encoding/">
  <s:Body>
    <u:GetTransportInfo xmlns:u="urn:schemas-upnp-org:service:AVTransport:1">
      <InstanceID>0</InstanceID>
    </u:GetTransportInfo>
  </s:Body>
</s:Envelope>
```

Each internal function uses `fetch` with POST method, `Content-Type: text/xml; charset="utf-8"`, and the appropriate SOAPAction header (with surrounding double quotes inside the header value string, matching Phase 1 pattern). Throw on non-ok HTTP response.

**Do NOT add timeouts or AbortController** — keep it simple, matching Phase 1 pattern. The `Promise.all` + try/catch with `null` return is sufficient.

**Consider DRY:** The speaker lookup + coordinator resolution + try/catch + state read pattern repeats across 7 command endpoints. A small helper function inside the plugin is acceptable to reduce repetition:

```typescript
async function handleCommand(
  fastify: FastifyInstance,
  id: string,
  reply: FastifyReply,
  label: string,
  execute: (speaker: SpeakerInfo, coordinator: SpeakerInfo) => Promise<void>,
) {
  const speaker = fastify.speakers.getById(id)
  if (!speaker) return reply.status(404).send({ error: 'Speaker not found', uuid: id })

  const coordinator = fastify.speakers.getCoordinator(id) ?? speaker

  try {
    await execute(speaker, coordinator)
    const state = await readSpeakerState(speaker.ip, coordinator.ip)
    fastify.log.info(`[sonos] ${label} → "${speaker.name}"`)
    return { ok: true, state }
  } catch (err) {
    const detail = err instanceof Error ? err.message : String(err)
    fastify.log.error(`[sonos] ${label} failed: ${detail}`)
    return reply.status(502).send({ error: 'Command failed', detail })
  }
}
```

This is discretionary — the executor may inline the pattern or use the helper. Either approach is acceptable as long as every route has correct routing (transport → coordinator, rendering → target).
